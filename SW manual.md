Что такое Service Worker

Service Worker — это специальный JavaScript-файл, который браузер запускает отдельно от страницы.

Проще говоря:
• обычный app.js работает внутри страницы;
• sw.js работает рядом с приложением, как фоновый помощник браузера.

Его главная задача — перехватывать сетевые запросы и решать:
• взять файл из интернета;
• взять файл из локального кэша;
• вернуть запасной ответ, если сети нет.

⸻

Очень простая аналогия

Представьте кафе.
• страница сайта — это зал, где сидят посетители;
• app.js — официант, который работает с клиентом;
• Service Worker — это кладовщик на складе.

Когда посетитель просит блюдо: 1. официант передаёт запрос; 2. кладовщик смотрит, есть ли это уже на складе; 3. если есть — отдаёт сразу; 4. если нет — заказывает снаружи; 5. если и снаружи нельзя достать — сообщает, что сейчас недоступно.

То есть Service Worker — это посредник между приложением и сетью.

⸻

Зачем он нужен

В учебном проекте он нужен для трёх вещей:

1. Офлайн-работа

Чтобы приложение могло открыться даже без интернета.

2. Быстрая загрузка

Если файл уже лежит в кэше, его не надо заново качать.

3. Управление ресурсами

Можно решать, какие файлы хранить локально, а какие — нет.

⸻

Что Service Worker НЕ делает

Это тоже важно.

Он:
• не рисует интерфейс;
• не работает напрямую с DOM;
• не хранит задачи в localStorage;
• не заменяет app.js.

То есть:
• app.js управляет интерфейсом и логикой страницы;
• sw.js управляет сетевыми запросами и кэшем.

⸻

Из каких частей состоит ваш Service Worker

В вашем примере он состоит из 4 основных частей: 1. имя кэша; 2. список файлов для кэширования; 3. обработчик install; 4. обработчик activate; 5. обработчик fetch.

Теперь по очереди.

⸻

1. CACHE_NAME

const CACHE_NAME = 'practice-13-14-cache-v3';

Это имя текущей версии кэша.

Зачем оно нужно

Чтобы браузер понимал:
• какой кэш актуальный;
• какие кэши старые и их можно удалить.

Почему есть v3

Потому что приложение меняется:
• добавили новые файлы;
• поменяли стили;
• изменили картинки;
• исправили логику.

Если версия не меняется, браузер может продолжать использовать старый кэш.

⸻

2. ASSETS_TO_CACHE

const ASSETS_TO_CACHE = [
'./',
'./index.html',
'./styles.css',
'./app.js',
'./manifest.json',
...
];

Это список файлов, которые нужно заранее сохранить локально.

Что здесь лежит

Это основные ресурсы приложения:
• HTML;
• CSS;
• JS;
• manifest;
• картинка;
• иконки.

Что это даёт

Когда пользователь впервые открывает приложение онлайн, Service Worker может заранее скачать эти файлы и положить их в кэш.

Потом, если интернет пропадёт, браузер всё равно сможет взять их локально.

⸻

3. Событие install

self.addEventListener('install', (event) => {
event.waitUntil(
caches.open(CACHE_NAME).then((cache) => cache.addAll(ASSETS_TO_CACHE))
);

self.skipWaiting();
});

Это первый важный этап жизни Service Worker.

Что означает install

Это момент, когда браузер:
• впервые установил Service Worker;
• или увидел его новую версию.

Что здесь происходит

Шаг 1

Открывается кэш:

caches.open(CACHE_NAME)

Если такого кэша ещё нет — он создаётся.

Шаг 2

В этот кэш кладутся все файлы из списка:

cache.addAll(ASSETS_TO_CACHE)

Шаг 3

event.waitUntil(...) говорит браузеру:

не считай установку завершённой, пока кэширование не закончится

То есть install должен не просто “запуститься”, а реально завершить сохранение файлов.

Шаг 4

self.skipWaiting() говорит:

новая версия Service Worker, не жди слишком долго, переходи к работе быстрее

⸻

4. Событие activate

self.addEventListener('activate', (event) => {
event.waitUntil(
caches.keys().then((cacheKeys) => {
return Promise.all(
cacheKeys
.filter((key) => key !== CACHE_NAME)
.map((oldKey) => caches.delete(oldKey))
);
})
);

self.clients.claim();
});

Это второй этап жизни Service Worker.

Что означает activate

Это момент, когда новый Service Worker стал активным.

Зачем нужен этот этап

Чтобы:
• удалить старые кэши;
• начать обслуживать приложение.

Что здесь происходит

Шаг 1

Получаем список всех кэшей:

caches.keys()

Шаг 2

Оставляем только старые, не равные текущему имени:

.filter((key) => key !== CACHE_NAME)

Шаг 3

Удаляем их:

caches.delete(oldKey)

Зачем это нужно

Иначе в браузере будут копиться:
• старые CSS;
• старые JS;
• старые картинки;
• старые версии приложения.

Это приведёт к путанице.

self.clients.claim()

Это говорит:

начни сразу управлять уже открытыми вкладками приложения

Без этого новый SW может начать работать не сразу.

⸻

5. Событие fetch

Это главная часть.

self.addEventListener('fetch', (event) => {
...
});

Что это такое

Каждый раз, когда страница запрашивает ресурс, Service Worker может этот запрос перехватить.

Например, страница просит:
• index.html
• styles.css
• app.js
• картинку
• иконку

Service Worker говорит:

подождите, я сам решу, откуда это взять

⸻

Почему есть проверка на GET

if (event.request.method !== 'GET') {
return;
}

Потому что здесь кэшируются только обычные запросы на получение ресурсов.

Почему не трогаем POST, PUT, DELETE

Потому что:
• это уже не просто загрузка файла;
• это запросы на изменение данных;
• кэшировать их бездумно нельзя.

Для учебного проекта это правильно и безопасно.

⸻

Как работает стратегия Cache First

Ваш код реализует стратегию Cache First.

Это значит:

Сначала ищем в кэше

caches.match(event.request)

Если ответ найден:
• сразу возвращаем его;
• в сеть уже не идём.

Если в кэше нет — идём в интернет

fetch(event.request)

Если интернет есть:
• получаем ресурс;
• возвращаем его.

Если и в кэше нет, и сети нет

Возвращаем запасной ответ:

new Response('Офлайн: ресурс недоступен ...')

⸻

Что такое кэш в этом контексте

Это не localStorage.

Это Cache Storage — отдельное браузерное хранилище для ответов сети.

Очень важно не путать:

localStorage

Хранит данные в формате строк:
• задачи;
• настройки;
• текст.

Cache Storage

Хранит сетевые ответы:
• HTML;
• CSS;
• JS;
• изображения;
• манифест.

⸻

Как работает весь сценарий по шагам

Первый запуск приложения 1. Пользователь открывает сайт. 2. Загружается index.html. 3. app.js регистрирует sw.js. 4. Service Worker устанавливается. 5. На этапе install кладёт основные файлы в кэш. 6. На этапе activate удаляет старые версии кэша. 7. После этого готов перехватывать запросы.

Следующий запуск

Когда пользователь снова открывает приложение: 1. браузер просит index.html; 2. Service Worker перехватывает запрос; 3. ищет index.html в кэше; 4. если файл есть — отдаёт его сразу; 5. то же самое для CSS, JS, картинок.

Если интернета нет

Тогда Service Worker всё равно может отдать файлы из кэша.
Поэтому оболочка приложения откроется офлайн.

⸻

Почему приложение открывается офлайн, а задачи не пропадают

Потому что тут работают два механизма одновременно.

Service Worker

Даёт:
• HTML;
• CSS;
• JS;
• картинки;
• иконки.

То есть саму оболочку приложения.

localStorage

Даёт:
• список задач;
• статусы выполнено / не выполнено;
• дату создания.

То есть пользовательские данные.

⸻

Из каких логических частей состоит работа SW

Если совсем по-простому, то Service Worker состоит из трёх смысловых блоков:

1. Подготовка

Что кэшировать и как назвать кэш.

Это:
• CACHE_NAME
• ASSETS_TO_CACHE

2. Установка и обновление

Что делать при первом подключении и при новой версии.

Это:
• install
• activate

3. Обработка запросов

Как вести себя, когда приложению нужен файл.

Это:
• fetch

⸻

Что значит “более продвинутые стратегии”

Сейчас у вас самая простая логика:
• если файл есть в кэше — берём его;
• если нет — идём в сеть.

Но бывают и другие стратегии.

Cache First

Сначала кэш, потом сеть.
Подходит для:
• картинок;
• иконок;
• статики.

Network First

Сначала сеть, потом кэш.
Подходит, когда важнее свежесть данных.

Stale While Revalidate

Сначала быстро отдаём старый кэш, а потом в фоне обновляем его из сети.

Это уже более продвинутый уровень, который вы и оставили студентам.

⸻

Самое главное, что надо понять

Service Worker — это не “магия PWA” и не “второй app.js”.

Это фоновый посредник, который:
• слушает события браузера;
• работает с кэшем;
• перехватывает запросы;
• помогает приложению работать быстрее и офлайн.

⸻

Ещё проще, в виде схемы

Пользователь открывает страницу
↓
Страница запрашивает HTML / CSS / JS / изображения
↓
Service Worker перехватывает запрос
↓
Проверяет кэш
├─ если файл есть → отдаёт из кэша
└─ если файла нет → идёт в сеть
├─ сеть есть → отдаёт из сети
└─ сети нет → отдаёт fallback
